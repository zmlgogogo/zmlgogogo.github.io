西门子开放式用户通讯 OUC 详解：TSEND_C / TRCV_C（S7-1200/1500）

讲解西门子 S7-1200/1500 的开放式用户通讯（OUC），用 TSEND_C / TRCV_C 实现与上位机、第三方设备的 TCP/ISO-on-TCP/UDP 自由通讯，涵盖连接参数 TCON、收发指令参数、数据收发处理与常见排查。

当 PLC 需要和上位机软件、视觉系统、扫码枪、第三方控制器等「非西门子」设备交换自定义数据时，Modbus 和 S7 通讯往往不够灵活。这时就轮到 开放式用户通讯（OUC，Open User Communication） 出场 —— 它让 PLC 直接基于 TCP/IP 收发自定义字节流，想发什么发什么。本文讲解 S7-1200/1500 用 TSEND_C / TRCV_C 实现 OUC 的完整方法。

一、OUC 基础#

1.1 什么是开放式用户通讯#

OUC 是西门子基于标准以太网传输层提供的「自由通讯」机制：PLC 自己组织报文内容，通过 TCP 连接收发原始字节数据，协议内容由用户自定义。

支持的连接类型：

类型 说明 适用
TCP 标准 TCP/IP 流 与上位机、第三方设备最常用
ISO-on-TCP 带 TSAP 寻址（RFC 1006） 西门子设备间、需 TSAP 区分
UDP 无连接数据报 广播/低开销场景

1.2 与其他通讯的区别#

方式 协议内容 对方要求
Modbus TCP 固定 Modbus 协议 对方支持 Modbus
S7 通讯 西门子私有 仅 SIMATIC 设备
OUC 用户自定义 任意支持 TCP/IP 的设备

OUC 的精髓是「自由」：报文格式你定，所以它能对接几乎任何以太网设备，但也意味着收发双方要约定好协议格式（帧头、长度、结束符、字节序等）。

1.3 核心指令#

指令 功能
TSEND_C 建立连接 + 发送数据（带连接管理，推荐）
TRCV_C 建立连接 + 接收数据（带连接管理，推荐）
TCON / TDISCON 单独建立 / 断开连接（精细控制时用）
TSEND / TRCV 仅收发（需配合 TCON 已建连接）

TSEND_C/TRCV_C 把连接管理和收发合二为一，最常用，本文以它们为主。

二、连接参数 TCON_IP_v4#

OUC 同样用 TCON_IP_v4（TCP/UDP）或 TADDR/TCON_QDN 等连接参数结构描述连接。在 TSEND_C/TRCV_C 指令上可直接「组态」生成，或手动在 DB 里建结构。

TCP 连接关键字段：

字段 说明
InterfaceId 本地以太网口硬件标识符
ID 连接 ID（1~4095），每条连接唯一
ConnectionType TCP=16#0B，UDP=16#13，ISO-on-TCP=16#11
ActiveEstablished 主动方 TRUE，被动方 FALSE
RemoteAddress 对方 IP（主动方填）
RemotePort 对方端口（主动方填，如 2000）
LocalPort 本地端口（被动方填，监听端口）

谁主动连谁就 ActiveEstablished := TRUE 并填对方 IP/端口；被动监听方 FALSE 并设 LocalPort。一般 PLC 连上位机时 PLC 主动；上位机连 PLC 时 PLC 被动。

三、TSEND_C —— 发送数据#

                TSEND_C
            ┌──────────────────────┐
   REQ ────►│ REQ              DONE ├──► 完成
  CONT ────►│ CONT             BUSY ├──► 忙
   LEN ────►│ LEN             ERROR ├──► 错误
 &CONN ────►│ CONNECT        STATUS ├──► 状态字
 &DATA ────►│ DATA                  │
            └──────────────────────┘
参数 说明
REQ 上升沿触发一次发送
CONT 1=建立并保持连接，0=断开。一般置 1 维持长连接
LEN 发送字节数（0=由 DATA 长度决定）
CONNECT 指向连接参数 TCON_IP_v4
DATA 发送数据区（指向 DB 数组或结构）
DONE/BUSY/ERROR/STATUS 完成/忙/错误/状态字

发送流程：CONT 保持 1 让连接建立 → 给 REQ 一个上升沿 → 等 DONE → 准备下一帧。

四、TRCV_C —— 接收数据#

                TRCV_C
            ┌──────────────────────┐
  EN_R ────►│ EN_R          RCVD_LEN├──► 实收字节数
  CONT ────►│ CONT              NDR ├──► 收到新数据
   LEN ────►│ LEN              BUSY ├──► 忙
 &CONN ────►│ CONNECT         ERROR├──► 错误
 &DATA ────►│ DATA           STATUS├──► 状态字
            └──────────────────────┘
参数 说明
EN_R 置 1 准备接收
CONT 1=保持连接
LEN 期望接收的字节数（0=用 ad-hoc 模式接收任意长度）
CONNECT 连接参数
DATA 接收数据存放区
NDR New Data Ready，收到新数据时为 1（一个扫描周期）
RCVD_LEN 实际接收字节数

接收模式：LEN 设固定值时按定长接收；设 0 时为 ad-hoc 模式，收多少给多少（适合不定长报文，常用于扫码枪等）。判断到 NDR=TRUE 时，从 DATA 取走数据并按 RCVD_LEN 解析。

五、实战示例：S7-1200 与上位机收发字符串#

场景：PLC（IP 192.168.0.1）主动连上位机服务器（IP 192.168.0.100，端口 2000），周期发送一帧数据，并接收上位机回复。

发送（TSEND_C）：

// 连接参数 conn1：ActiveEstablished=TRUE, RemoteAddress=192.168.0.100, RemotePort=2000, ID=1
"TSEND_C_DB"(REQ     := sendTrig,     // 上升沿触发发送
             CONT    := TRUE,         // 保持连接
             LEN     := 20,           // 发 20 字节
             CONNECT := "conn".conn1,
             DATA    := "sendBuf");   // Array[0..19] of Byte
IF "TSEND_C_DB".DONE THEN
    // 发送成功
    ;
END_IF;

接收（TRCV_C，用另一条 ID 或同连接）：

"TRCV_C_DB"(EN_R    := TRUE,          // 持续准备接收
            CONT    := TRUE,
            LEN     := 0,             // ad-hoc，不定长
            CONNECT := "conn".conn1,
            DATA    := "rcvBuf");
IF "TRCV_C_DB".NDR THEN
    // rcvBuf 中是新数据，长度 = "TRCV_C_DB".RCVD_LEN，按约定协议解析
    ;
END_IF;

同一条连接可同时收发（全双工 TCP）：用同一个 CONNECT/ID，TSEND_C 与 TRCV_C 各自背景 DB。也可一个方向用一条连接。

六、协议设计与数据处理要点#

OUC 自由度高，所以双方必须约定报文格式，否则收到一堆字节无法解析：

  1. 帧定界：约定帧头 + 长度字段，或结束符（如 \r\n），TCP 是字节流不保证一帧一次到达。
  2. 粘包/拆包：TCP 可能把多帧合并或一帧分多次到。接收端要按长度/定界符自行组帧，不能假设一次 NDR = 一完整帧。
  3. 字节序：与非西门子设备交换 16/32 位数值时注意大小端，必要时做字节交换。
  4. 字符串处理：用 Strg_TO_Chars / Chars_TO_Strg 在 String 与字节数组间转换。

七、常见问题与排查#

STATUS / 现象 含义 排查方向
0x7000 无作业（空闲） 正常，REQ 未触发
0x7001/0x7002 建立连接/处理中 正常过渡态
连接建立不上 IP/端口错、对方未监听、主被动设反 ping 测通、确认对方端口开放、核对 ActiveEstablished
0x8187 CONNECT 参数无效 检查 TCON_IP_v4 字段、ID 冲突
收到数据乱/不全 粘包拆包、长度处理错 按帧头+长度组帧，别假设一次收全
数值高低字节反 字节序差异 做字节/字交换
连接频繁断 CONT 置 0、对方主动断、网络 CONT 保持 1、查网络与对方

排查口诀#

  1. 先通网络：ping 对方，确认端口开放、防火墙放行。
  2. 定主被动：谁主动谁 ActiveEstablished=TRUE 填对方 IP/端口；被动方设 LocalPort。
  3. ID 唯一：每条连接 ID 不重复。
  4. 保持连接：CONT/EN_R 维持长连接，REQ 边沿触发发送。
  5. 组帧解析：按约定帧格式处理粘包/拆包与字节序。

八、要点速查#

环节 关键点
适用 对接上位机、视觉、扫码枪、第三方设备自定义协议
连接类型 TCP（最常用）/ ISO-on-TCP / UDP
核心指令 TSEND_C 发 / TRCV_C 收（带连接管理）
连接参数 TCON_IP_v4，ID 唯一、主被动、IP/端口
触发 REQ 上升沿发送；EN_R 持续接收
接收 LEN=0 为 ad-hoc 不定长；判 NDR 取 RCVD_LEN
难点 自定义协议组帧 + 粘包拆包 + 字节序

结语#

开放式用户通讯是 PLC 对接「非西门子世界」的万能钥匙：基于标准 TCP/IP，用 TSEND_C/TRCV_C 收发自定义字节流，能连上几乎任何以太网设备。它和前面讲的 Modbus、S7 通讯一起，构成了西门子 PLC 通讯的完整拼图 —— 标准设备走 Modbus，西门子之间走 S7，自定义协议走 OUC。用好 OUC 的关键不在指令本身（参数很简单），而在和对方约定清楚报文格式、处理好 TCP 的粘包拆包。把这一点想透，PLC 与上位机、视觉、机器人之间的数据交换就能稳定可靠。
