S7 通讯（S7 Communication）是西门子私有的、专门用于 SIMATIC 设备之间 数据交换的协议，运行在工业以太网之上。相比 Modbus 这类开放协议，S7 通讯与西门子硬件深度集成、配置简单、传输高效，是 S7-300/400/1200/1500 之间互联数据的首选方式。本文聚焦 S7-1200 / S7-1500（TIA Portal） 平台，讲解 S7 通讯的两类实现：单边通讯（PUT/GET） 与 双边通讯（BSEND/BRCV）。

一、S7 通讯基础#

1.1 什么是 S7 通讯#

S7 通讯基于 ISO-on-TCP（RFC 1006），使用 TCP 端口 102，是西门子设备间的「原生语言」。它的核心特点：

  • 专用于 SIMATIC 设备之间（S7-1200/1500/300/400），不与第三方通用设备通讯（那是 Modbus/OUC 的活）。
  • 通过 S7 连接 描述通讯双方，连接在「设备组态 → 网络视图」里建立。
  • 数据直接读写对方的 DB 块，无需对方编程配合（单边模式）。

1.2 单边通讯 vs 双边通讯#

这是理解 S7 通讯最关键的一对概念：

对比项 单边通讯（PUT/GET） 双边通讯（BSEND/BRCV）
指令 PUT（写）/ GET（读） BSEND（发）/ BRCV（收）
编程位置 只在一侧（客户端）编程 双方都要编程，成对配合
对方角色 服务器，被动被读写，无需写程序 主动参与，需调用接收/发送指令
数据量 较小 较大（最高约 64KB）
握手确认 无应用层握手 有，发送/接收成对确认，可靠性高
适用平台 S7-1200/1500/300/400 S7-1500/400（1200 不支持 BSEND/BRCV 作为通讯方）
典型场景 1200 读写另一台 1200/1500 的 DB 1500↔1500/400 之间大数据、高可靠传输

S7-1200 只支持单边的 PUT/GET（且需开启权限），不支持 BSEND/BRCV。如果通讯双方都是 1200，只能用 PUT/GET。BSEND/BRCV 主要用于 S7-1500 与 S7-1500/400 之间。

二、单边通讯：PUT / GET（S7-1200/1500）#

PUT/GET 是最常用的 S7 通讯方式：只在「客户端」一侧编写程序，主动读取（GET）或写入（PUT）对方「服务器」的数据块，服务器一侧无需任何通讯程序。

指令位于指令树 通信 → S7 通信。

2.1 前置组态：建立 S7 连接#

  1. 进入 网络视图，选择「连接」模式，连接类型选 S7 连接。
  2. 从客户端 CPU 的以太网口拖拽到服务器 CPU，建立一条 S7 连接，系统分配 连接 ID（如 W#16#100）。
  3. 记下连接 ID 和伙伴的接口信息，PUT/GET 指令的 ID 参数要用到。

2.2 关键前提：开启「PUT/GET 访问权限」#

这是新手最常卡住的一步。 作为服务器的 S7-1200/1500 CPU，默认禁止外部通过 PUT/GET 访问，必须手动开启：

选中服务器 CPU → 属性 → 防护与安全（Protection & Security） → 连接机制 → 勾选 「允许来自远程对象的 PUT/GET 通信访问」。

不勾这一项，客户端的 PUT/GET 会一直报错（连接拒绝），且报错信息不直观，极易误判。

2.3 GET —— 读取伙伴数据#

                  GET
            ┌──────────────────┐
   REQ ────►│ REQ        NDR    ├──► 收到新数据
W#16#100───►│ ID         ERROR  ├──► 错误
 &ADDR ────►│ ADDR_1     STATUS ├──► 状态字
 &RD ──────►│ RD_1              │
            └──────────────────┘
参数 说明
REQ 上升沿触发一次读取
ID S7 连接 ID（与网络组态里的一致，如 W#16#100）
ADDR_1 要读取的伙伴侧地址区（指针，指向对方的 DB，如 P#DB10.DBX0.0 BYTE 20）
RD_1 读到的数据存放在本地的位置
NDR New Data Ready，读取成功置 1
ERROR / STATUS 错误标志 / 状态字

可用 ADDR_1ADDR_4 / RD_1RD_4 一次读取最多 4 个区域。

2.4 PUT —— 写入伙伴数据#

                  PUT
            ┌──────────────────┐
   REQ ────►│ REQ        DONE   ├──► 完成
W#16#100───►│ ID         ERROR  ├──► 错误
 &ADDR ────►│ ADDR_1     STATUS ├──► 状态字
 &SD ──────►│ SD_1              │
            └──────────────────┘
参数 说明
REQ 上升沿触发一次写入
ID S7 连接 ID
ADDR_1 要写入的伙伴侧地址区（指向对方 DB）
SD_1 本地要发送的数据源
DONE 写入成功置 1

同样支持 ADDR_1ADDR_4 / SD_1SD_4 一次最多写 4 个区域。

2.5 关于「优化的块访问」#

被读写的 DB 如果在客户端用 绝对地址指针（P#DBx.DBXy.z） 访问，那么伙伴侧（服务器）被访问的 DB 必须设为「非优化访问」（取消勾选「优化的块访问」），否则地址无法对应。这是 PUT/GET 跨 CPU 访问的常见约束。

2.6 PUT/GET 轮询要点#

  • PUT 与 GET 都用 REQ 上升沿触发，不能常 1。
  • 多个 PUT/GET 需要轮流执行时，用各自的 DONE/NDR 做状态机串行触发。
  • 同一条 S7 连接（同一 ID）上的请求会排队，避免并发冲突。

三、双边通讯：BSEND / BRCV（S7-1500）#

当需要在 S7-1500 之间传输大块数据且要求高可靠性时，使用双边通讯。它要求通讯双方都编程：发送方调用 BSEND，接收方调用 BRCV，二者通过相同的 R_ID 配对。

指令位于 通信 → S7 通信。

3.1 BSEND —— 发送数据（发送方）#

                 BSEND
            ┌──────────────────┐
   REQ ────►│ REQ        DONE   ├──► 完成
     R ────►│ R          ERROR  ├──► 错误
W#16#100───►│ ID         STATUS ├──► 状态字
   1 ──────►│ R_ID              │
 &SD ──────►│ SD_1              │
 LEN ──────►│ LEN               │
            └──────────────────┘
参数 说明
REQ 上升沿触发发送
R 复位，置 1 中止当前发送
ID S7 连接 ID
R_ID 通讯任务标识，必须与接收方 BRCV 的 R_ID 相同，用于配对
SD_1 要发送的数据区（指针）
LEN 实际发送的数据长度（字节）
DONE 接收方确认收到后置 1

3.2 BRCV —— 接收数据（接收方）#

                 BRCV
            ┌──────────────────┐
    EN_R ──►│ EN_R       NDR    ├──► 收到新数据
W#16#100───►│ ID         ERROR  ├──► 错误
   1 ──────►│ R_ID       STATUS ├──► 状态字
 &RD ──────►│ RD_1       LEN    ├──► 实收长度
            └──────────────────┘
参数 说明
EN_R 置 1 表示准备好接收
ID S7 连接 ID
R_ID 必须与发送方 BSEND 的 R_ID 相同
RD_1 接收数据存放区
NDR 收到新数据置 1
LEN 实际接收到的字节数

3.3 双边通讯要点#

  1. R_ID 必须成对一致：BSEND 与 BRCV 通过相同的 R_ID 建立逻辑通道，一条连接上可有多个不同 R_ID 的通道。
  2. 双方都要编程：发送方有 BSEND，接收方必须有对应的 BRCV 在运行，否则发送方一直 BUSY/超时。
  3. 数据量大：BSEND/BRCV 单次可传输最高约 64KB，远超 PUT/GET，适合配方、批量数据。
  4. 有握手确认：BSEND 的 DONE 在接收方真正收到后才置 1，可靠性高于 PUT/GET。

四、实战示例：S7-1200 读写另一台 S7-1200#

场景：PLC-A（客户端，IP 192.168.0.1）需要读取 PLC-B（服务器，IP 192.168.0.2）DB10 的 20 字节，并向 PLC-B 的 DB11 写 10 字节。

服务器侧（PLC-B）准备：

  1. 在「防护与安全」中勾选 「允许 PUT/GET 通信访问」。
  2. DB10、DB11 设为非优化访问。
  3. 无需编写任何通讯程序。

客户端侧（PLC-A）编程：

  1. 网络视图建立 A→B 的 S7 连接，记下 ID（假设 W#16#100）。
  2. OB1 中调用 GET 与 PUT：
// 每 100ms 一个脉冲 trigGet 读取 PLC-B 的 DB10
"GET_DB"(REQ    := trigGet,
         ID     := W#16#100,
         ADDR_1 := P#DB10.DBX0.0 BYTE 20,   // 伙伴 PLC-B 的 DB10
         RD_1   := "rdBuf");                 // 本地接收缓冲
IF "GET_DB".NDR THEN
    // rdBuf 已更新
    ;
END_IF;

// 读完成后触发写 trigPut，向 PLC-B 的 DB11 写 10 字节
"PUT_DB"(REQ    := trigPut,
         ID     := W#16#100,
         ADDR_1 := P#DB11.DBX0.0 BYTE 10,   // 伙伴 PLC-B 的 DB11
         SD_1   := "wrBuf");                 // 本地数据源
IF "PUT_DB".DONE THEN
    ;
END_IF;

五、常见问题与排查#

现象 可能原因 排查方向
GET/PUT 报错、连接拒绝 服务器未开启「PUT/GET 访问权限」 服务器 CPU → 防护与安全 → 勾选允许 PUT/GET
地址访问失败 / 数据错乱 被访问 DB 是「优化访问」 把伙伴侧 DB 改为非优化访问
连接建立不上 IP 不通、连接 ID 不匹配、子网不一致 ping 测通、核对网络组态里的连接 ID
BSEND 一直 BUSY 不完成 接收方无 BRCV 运行 或 R_ID 不一致 确认对方 BRCV 在跑、R_ID 两侧相同
数据只更新一次 REQ/EN_R 用了常 1 或未做边沿 GET/PUT 用上升沿触发，BRCV 的 EN_R 保持使能
S7-1200 用 BSEND 失败 1200 不支持 BSEND/BRCV 改用 PUT/GET

排查口诀#

  1. 先通网络：ping 双方 IP，确认同子网、端口 102 可达。
  2. 再开权限（PUT/GET 必查）：服务器侧勾选「允许 PUT/GET 访问」。
  3. 查 DB 属性：被访问 DB 设为非优化访问。
  4. 核连接参数：连接 ID 一致；双边通讯核对 R_ID 成对相同。
  5. 看时序：REQ 边沿触发，多请求用 DONE/NDR 串行轮询。

六、S7 通讯方式对照#

维度 PUT / GET（单边） BSEND / BRCV（双边）
编程侧 仅客户端一侧 发送方 + 接收方双方
对方配合 不需编程（仅开权限、设非优化 DB） 需编写对应收/发指令
数据量 小 大（约 64KB）
可靠性 无应用层握手 成对握手确认
支持平台 1200/1500/300/400 1500/400（1200 不支持）
适用场景 跨 CPU 读写少量 DB 数据 大数据、配方、高可靠传输

结语#

S7 通讯是西门子设备之间最自然、最高效的互联方式。日常工程中，PUT/GET 单边通讯覆盖了绝大多数需求：只要在服务器侧勾选「允许 PUT/GET 访问」、把被访问的 DB 设为非优化，再在客户端用 GET/PUT 读写即可，对方甚至不用写一行通讯代码。只有当通讯双方都是 S7-1500/400、且需要传输大块数据并要求高可靠握手时，才用 BSEND/BRCV 双边通讯。记住两条「保命」要点 —— PUT/GET 必开访问权限、被访问 DB 必设非优化 —— 就能避开 S7 通讯里九成的坑。
